# The LiteLLM Decision and the OTel-Pure Choice

## The story in one sentence
A March 2026 supply chain attack on LiteLLM removed it from the
project's candidate list entirely — and the resulting decision, to
skip the LLM-call proxy layer altogether and talk directly to vLLM's
OpenAI-compatible REST API with an OTel-pure observability stack,
turned out to be exactly the right architecture for a
Federal-credible edge deployment.

## Why this is its own entry

The broader observability story is in
[`journey/03-observability`](03-observability.md). This entry
focuses specifically on the **LiteLLM supply chain decision** and
why it is load-bearing for Federal customers. It's worth
documenting separately because "we decided not to use a component
that would otherwise have been the obvious choice" deserves a
clear explanation, and the reason — supply chain compromise — is
relevant to any team making architecture decisions in 2026 for
regulated environments.

## What LiteLLM is and why it's usually the default

[LiteLLM](https://github.com/BerriAI/litellm) is an open-source
proxy that sits between your application and any LLM provider. It
accepts OpenAI-compatible API calls and forwards them to whichever
backend you configure — OpenAI, Anthropic, Azure, a local vLLM
endpoint, etc. For teams building on top of multiple model
providers, LiteLLM is extremely convenient: you write code against
one API, and you can swap backends by changing a config.

For an LLM observability system like Langfuse, LiteLLM is often the
recommended integration point: instead of instrumenting every
individual LLM client library, you route all traffic through
LiteLLM, and LiteLLM handles the OpenInference trace export. One
integration, every call captured.

In ordinary circumstances, LiteLLM would have been the default
choice for this project too. The PRD mentioned it. Early design
sketches assumed it. We would have proxied the harness's vLLM
calls through LiteLLM and routed the traces to a local Langfuse
instance.

## What changed in March 2026

In early March 2026, LiteLLM versions 1.82.7 and 1.82.8 were
published to PyPI containing malicious code. The compromise was
traced back to a poisoned dependency in one of LiteLLM's own
build-time tools — ironically, a security-scanning tool (Trivy)
that had itself been compromised upstream. The malicious versions
were available on PyPI for a period of hours before being detected
and removed.

The specifics of the compromise matter less than the structural
fact: **LiteLLM became, for a moment, an attack surface** — a
component that could, in a compromised version, exfiltrate LLM
traffic or inject responses. For a project targeting Federal
audiences, where every dependency has to be defensible to a
skeptical CISO, this is a hard constraint.

Important caveats:

- The LiteLLM project responded quickly. The malicious versions
  were removed; fixes were shipped; the incident was disclosed
  honestly.
- This is not unique to LiteLLM. **Any package ecosystem can be
  compromised.** Python, npm, Rust crates, Go modules — all have
  had supply chain incidents in recent years. LiteLLM's
  misfortune was that it was a high-value target because of its
  position in the LLM stack.
- The responsible thing for us to do is not to blame LiteLLM but
  to recognize that **the proxy layer itself is a risk to
  consolidate**, regardless of which specific project provides it.

The decision we made wasn't "LiteLLM is bad." The decision was
"having a proxy layer in the critical path of every LLM call is a
risk that buys us less than we thought, and the project can avoid
that risk entirely by talking to vLLM directly."

## The alternative: OTel-pure and direct vLLM calls

Once we decided not to use a proxy, the architecture clarified:

### 1. Direct vLLM calls from the harness

The harness talks to vLLM's OpenAI-compatible REST endpoint
directly via `httpx`. No proxy layer, no abstraction wrapper, no
additional process in the path. vLLM is the only LLM-facing
service on the box, and the harness is the only LLM client.

This has a few side benefits beyond supply chain reduction:

- **Fewer processes to run.** One less systemd unit, one less
  thing to monitor, one less thing to restart.
- **Fewer network hops per call.** Harness → vLLM is a single
  HTTP call over localhost. No extra hop through a proxy.
- **Simpler failure modes.** If something goes wrong with an LLM
  call, it's either in the harness code or in vLLM. There's no
  proxy layer to troubleshoot.
- **Clearer operational picture.** The logs for an LLM call come
  from two places (harness + vLLM), not three. Debugging is
  faster.

### 2. OTel-pure observability

With no proxy to do the tracing for us, we had to instrument the
harness directly. The approach:

- The harness uses the OpenTelemetry Python SDK to create spans
  around each LLM call, each tool invocation, each Ralph loop
  iteration.
- Spans are decorated with GenAI semantic conventions
  (`gen_ai.request.model`, `gen_ai.usage.prompt_tokens`, etc.)
  so downstream consumers can reason about LLM-specific metrics
  in a standard way.
- The collector (running as a separate process on the same host)
  receives the spans and exports them to Jaeger for distributed
  tracing, Prometheus for metrics, and Grafana for dashboards.
- No Langfuse, no LiteLLM, no proprietary observability stack.
  Just the CNCF-standard OTel tools.

The stack is: **OpenTelemetry Collector → Jaeger + Prometheus +
Grafana**, all open source, all standards-based, all
well-understood in Federal environments.

### 3. Redundant structured run logging

Alongside the OTel stack, the harness also writes a structured
JSONL event log per run (see
[`journey/12.5-structured-run-logger`](12.5-structured-run-logger.md)).
This duplication is intentional:

- The OTel stack is the **live operational view** — what's
  happening right now, distributed traces, metrics over time.
- The JSONL log is the **authoritative per-run record** — a
  complete, self-contained, replayable history of a single run.

The two serve different questions and can each stand alone. If
the OTel stack is unavailable (stopped, corrupted, behind a
firewall), the JSONL log still captures everything needed for
post-run analysis. If the JSONL log is lost, the OTel stack
still shows the live view. Neither depends on the other.

## The tradeoffs this decision makes

Being honest about what we gave up by not using LiteLLM:

- **Multi-backend portability.** If a future version of this
  project wanted to run against OpenAI's API instead of local
  vLLM, the harness would need a small adapter — it currently
  has one backend assumed. LiteLLM would have made the swap
  trivial. This is acceptable for a project whose whole point
  is local-first, air-gap-capable, edge-deployed inference.
- **Pre-built LLM observability views.** Langfuse's UI is very
  good at "show me all my prompts, their token costs, their
  latency distributions." We get the same data through the OTel
  stack but the dashboards are ours to build. This is more
  work, but the dashboards we end up with are tailored to the
  reflexion-loop use case in a way the generic LLM observability
  tool wouldn't be.
- **Community knowledge.** When something breaks, there's less
  stack overflow content for "direct vLLM + custom OTel" than
  for "LiteLLM + Langfuse." We trade community support for
  architectural simplicity. The project is small enough for us
  to understand every piece directly, so this trade is fine.

## Why this lands for Federal customers

The specific framing that matters in a Federal conversation:

1. **Every LLM call is in the direct blast radius of the
   harness.** There is no third-party process that can see our
   prompts, responses, or metadata. The data stays between the
   harness and vLLM, both of which are components we control and
   audit directly.

2. **Every dependency in the critical path is auditable.** vLLM
   is open source, widely used, well-maintained, and on an
   OS-level distribution path (NVIDIA ships it, Red Hat ships
   RHAIIS on top of it). The OTel stack is CNCF-governed,
   standards-based, and ubiquitous. No proprietary APIs, no
   vendor lock-in, no "phone home" behavior.

3. **The supply chain surface is smaller.** Fewer dependencies
   in the critical path mean fewer things to pin, fewer things
   to hash-verify, fewer things to review on every upgrade. For
   any team building a Federal-audited pipeline (SBOM + sigstore
   + scorecard), a smaller dependency tree is always better.

4. **The incident that prompted this decision is a teaching
   moment.** When explaining the architecture to a Federal
   evaluator, the LiteLLM decision is a concrete, dated,
   verifiable event — not a hypothetical risk. "We made this
   choice because of a March 2026 supply chain incident; the
   alternative we picked is simpler and has less surface area"
   is a much stronger argument than "we prefer fewer
   dependencies in principle."

## What we learned

1. **The "obvious" choice is not always the right choice for a
   specific audience.** LiteLLM is obviously the right choice
   for most teams most of the time. For a Federal-audience
   project where supply chain is a first-class concern, it
   wasn't. The lesson is: read the room, then architect.

2. **Supply chain risk is a real architectural driver in 2026.**
   "Which libraries we use" is now part of architecture
   discussion in a way it wasn't five years ago. Treating it as
   a serious input to design decisions — rather than an
   afterthought — produces better architectures for regulated
   environments.

3. **Standards-based observability beats vendor observability for
   Federal audiences.** OTel + Jaeger + Prometheus + Grafana is
   unexciting, but "unexciting" is the highest compliment a
   Federal evaluator can give a software stack. It's well known,
   well understood, well supported, and well audited. Vendor
   observability tools can be better UX but harder to defend.

4. **Having two independent observability paths is good
   hygiene.** The OTel stack and the JSONL run log each can
   answer the questions the other can. If one stops working for
   any reason, the other still tells the story. This is cheap
   defense in depth and worth the slight duplication.
