# The Structured Run Logger: JSONL as the Event Format

## The story in one sentence
Early in Phase 5 we made what felt like a boring plumbing decision
— emit a single structured JSONL event stream per run with
`{timestamp, elapsed_s, event_type, agent, iteration, data,
gpu_state}` records — and it turned out to be one of the
highest-leverage decisions in the whole project, because that
event stream became the backbone for the dashboard, the replay
mode, the post-run forensics, and every piece of automated test
instrumentation.

## Why this is worth its own entry

Structured logging is a "fine, yes, obviously" decision that nobody
writes up because it seems too pedestrian to bother. But the shape
of the event record, the decision to write one record per event
(not one per turn or per attempt), and the decision to include
everything needed for replay in each record turned out to be
load-bearing in a way we didn't anticipate. This entry exists so
the "obvious" decision is documented explicitly, because it wasn't
actually obvious — several less-useful shapes were possible.

## The event record shape

Every event in a run is a single JSON object on its own line:

```json
{
  "timestamp": "2026-04-12T00:50:30.123456+00:00",
  "elapsed_s": 0.1,
  "event_type": "iteration_start",
  "agent": "system",
  "iteration": 1,
  "data": {
    "iteration": 1,
    "failing": 270,
    "remediated": 0,
    "escalated": 0,
    "rules_per_hour": 0.0,
    "banned_patterns": 0,
    "...": "..."
  },
  "gpu_state": [
    {"index": 0, "name": "NVIDIA L4", "memory_used_mib": 21149, ...},
    {"index": 1, ...},
    {"index": 2, ...},
    {"index": 3, ...}
  ]
}
```

**Design decisions baked into this shape:**

1. **JSONL, not JSON.** One line per event so the file can be
   streamed, tailed, and incrementally parsed. A run produces
   thousands of events over hours; the dashboard tails the file
   to follow a live run; post-run analysis iterates
   line-by-line without loading everything into memory. A single
   JSON array would have been simpler to write but unusable as
   a live stream.

2. **`elapsed_s` alongside `timestamp`.** Timestamps let you
   correlate with wall-clock events; elapsed seconds let you
   read the run as a narrative without doing math. Both are
   cheap to write; having both makes replay and analysis code
   simpler.

3. **`event_type` as a string, not an enum.** An enum is
   tempting but it creates a backward-compat headache whenever
   a new event type is added. Strings let us add event types
   freely without breaking downstream consumers, and the
   taxonomy of event types is documented in the code that
   produces them, not in a shared type definition that has to
   be synchronized across consumers.

4. **`data` as a free-form dict.** Each event type has its own
   schema. Rather than force everything into a union type, we
   let each event carry its own payload. The tradeoff is that
   consumers have to know which fields to expect for which
   event types — which we accept because the alternative
   (highly-structured, tightly-coupled schemas) would make it
   painful to add new fields.

5. **`gpu_state` optionally inline.** Some events include a
   snapshot of GPU state at the time of the event. Including it
   inline means the record is self-contained for replay — the
   dashboard can show what the GPUs looked like at any moment
   without joining against a separate time series. The cost is
   record size; for events that care about GPU state
   (heartbeats, attempt_start, revert), it's worth the size.

## Event types we emit

A partial list, as of the v3 architecture:

| Event type | When it fires | Key fields |
|---|---|---|
| `run_start` | Run begins | config snapshot, skill name |
| `skill_manifest` | Run begins | UI labels, outcome taxonomy, work item name |
| `snapshot_preflight` | Run begins | baseline exists, progress cleared |
| `scan_complete` | Initial STIG scan done | total failing rules count |
| `iteration_start` | Each outer iteration | counts, rates, memory tier sizes (heartbeat) |
| `rule_selected` | Architect picks a rule | rule_id, category, iteration, time budget |
| `attempt_start` | Each inner attempt | rule_id, attempt number, rule elapsed |
| `prompt_assembled` | Before each LLM call | budget, used tokens, sections included/dropped |
| `agent_response` | After each LLM turn | text, tokens, timing, tool_calls, capped flag |
| `tool_call` / `tool_result` | Each tool invocation | tool name, args, result |
| `evaluation` | Deterministic evaluation run | passed, health_ok, rule_ok, journal_clean |
| `remediated` | Rule succeeded | rule_id, attempt, wall time, phase timing, snapshot saved |
| `post_mortem` | Before a revert | full diagnostic forensics |
| `revert` | After a failed attempt | method (snapshot_restore), restore_ok, post_restore_healthy |
| `reflection` | Reflector analysis | text, attempt, ban count, plateau flag, phase timing |
| `ban_added` | Reflector adds a banned pattern | pattern, total bans |
| `architect_reengaged` | Architect re-engagement fires | trigger, verdict, full response |
| `escalated` | Rule escalation | reason (time_budget / retry_ceiling / architect_preemptive) |
| `rule_complete` | Each rule finishes (either way) | outcome, attempts, wall time, plateau, reengagement count |
| `tool_call_capped` | Harness intercepts a second tool call | count, max, tool |
| `tool_error` | External tool raises | phase, error message |
| `environment_unrecoverable` | Post-restore probe fails | detail, restore detail |
| `error` | Uncaught error in a turn | error message |
| `run_complete` | Run ends | summary counts |

Each event type has a consumer. Nothing is emitted "just in case."
If you can't name who reads the event, don't emit it.

## What the event stream unlocked

Once the logger was in place, several downstream pieces became
possible that we didn't initially plan for:

### The live dashboard
The dashboard is a Next.js app that reads events via a FastAPI
SSE endpoint. The backend tails the JSONL file; when new events
arrive, it pushes them to connected clients. The dashboard has
no direct coupling to the harness code — it just consumes
events. This separation means the harness can be refactored
freely without breaking the dashboard.

### Replay mode
The same dashboard can replay any historical run by reading the
JSONL file from disk and streaming it to the UI with a
configurable speed multiplier. No special-case replay logic in
the frontend — it's the same code path as live mode, just with a
file iterator instead of a live tail. This is how the demo can
show "here's what the v2 overnight run looked like" next to
"here's what the v3 run looks like" without any extra code.

### Post-run forensics
When a run finishes (or is killed), the entire JSONL file is a
complete, self-contained record. Any question we want to ask
after the fact — "how many reflections cited sudo?" "what was the
time distribution of escalated rules?" "did the plateau detector
fire correctly?" — is a small Python script over the file. No
database to query, no log aggregator to set up, no structured
event schema to maintain across services. The whole analysis
tier is `with open(...) as f: for line in f: event = json.loads(line)`.
This turned out to be the backbone of the overnight-run
postmortem; see [`journey/14`](14-overnight-run-findings.md).

### Test instrumentation
The Tier 5 integration test launches a constrained ralph run
against a single rule and then asserts properties of the
resulting JSONL — *"scan_complete, iteration_start, rule_selected,
attempt_start, agent_response, prompt_assembled, rule_complete
all fire in order, and rule_complete contains the expected
fields."* The test knows nothing about the harness internals; it
reads the event stream as ground truth. This is a much more
stable kind of test than one that patches internal state,
because the event stream is the external contract.

### Frontend time-travel debugging
Because every event has `elapsed_s` and the dashboard reads the
stream in order, you can pause the replay at any point and see
exactly what the harness knew at that moment. When we were
debugging a reflector plateau issue in the overnight run, we
replayed the run up to the moment of the first plateau and
looked at the exact reflections that had accumulated — a kind of
post-hoc pair-debugging where the run is the third party.

## The boring decision that mattered most

The "one structured record per event to one JSONL file" decision
is, in retrospect, the most important piece of infrastructure in
the project. Not because the JSON format is clever — it isn't —
but because **every downstream capability assumes this format
exists**, and if the format didn't exist, each downstream piece
would have to build its own ad-hoc logging.

If you are building a long-running agentic system and you don't
yet have a structured event log, stop and build one. Everything
downstream gets easier. Don't overthink the schema — start with
`{timestamp, event_type, data}` and iterate. The upgrade path is
easy because consumers just pattern-match on `event_type` and
ignore fields they don't know about.

## What we learned

1. **The event stream is the project's external contract.**
   Everything else — dashboard, replay, tests, postmortems,
   training data for future work — consumes it. Design the
   event schema with the assumption that someone you haven't
   met will try to extract information from it six months from
   now.

2. **Self-contained records beat normalized records.** Including
   GPU state inline on events that care about it, even though
   it's redundant, makes every downstream consumer simpler. The
   storage cost is negligible compared to the simplicity win.

3. **JSONL is the right format for a tailed-file use case.**
   Not JSON arrays (can't incrementally parse). Not CSV (no
   nested structures). Not protobuf (opaque to inspection).
   JSONL is ubiquitous, human-readable, streamable, and every
   language can parse it.

4. **Never emit an event you can't name a consumer for.** Every
   event type in our current schema has at least one thing that
   reads it. If we ever add an event type without a consumer,
   it becomes dead weight — growth in the log file without
   value.
